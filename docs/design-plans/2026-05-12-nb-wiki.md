# qDocent Notebook + Wiki Design

**Date:** 2026-05-12
**Slug:** `nb-wiki`
**Status:** Approved for implementation

---

## Summary

qDocent Notebook + Wiki adds a first-class organizational layer — the **Notebook** — on top of the existing RAG pipeline. Each Notebook maps 1:1 to an R2R collection, so document ingestion and retrieval are scoped natively: the API stores notebook metadata and document membership in SQLite (the source of truth), then mirrors that membership to an R2R collection by injecting a `collection_ids` filter into every retrieval call. The root `/ask` and `/conversations` routes stay untouched for unscoped requests, and an idempotent startup migration assigns all pre-existing documents to a "Default Notebook" so the upgrade is non-breaking.

Wiki generation runs as a two-step LLM pipeline inside a FastAPI `BackgroundTask`: one structure call to Gemini produces an XML outline of sections and pages, then N page calls execute in parallel via `asyncio.gather`, each retrieving the top-K chunks scoped to the page's source documents. Progress is tracked in a `wiki_jobs` SQLite table and polled by the frontend. Conversation persistence adds a thin SQLite metadata layer (notebook, auto-title, R2R conversation ID) without disturbing R2R's own message storage. Delivery is split into two implementation runs: phases 1–4 build the notebook model end-to-end, and phases 5–8 layer wiki generation and conversation history on top. LangGraph is explicitly out of scope for this design.

## Definition of Done

1. A user can create a **Notebook**, upload documents to it, and all RAG queries are scoped to that notebook via R2R collections.
2. After uploading, a **"Generate Wiki"** button triggers two-step generation (structure XML → parallel page synthesis) producing a navigable wiki with Mermaid diagrams and source citations.
3. A **wiki viewer** at `/notebooks/[id]/wiki/[slug]` renders pages with left tree nav, citation badges, and "Ask about this page" scoped chat.
4. **Conversations persist** to SQLite per notebook; a `/conversations` page lists them with auto-generated titles, dates, and notebook filter.
5. **Existing documents auto-migrate** to a "Default Notebook" on first boot after upgrade.
6. **8 implementation phases** delivered in two runs: phases 1–4 (Notebook model) then phases 5–8 (Wiki + Conversations).

## Glossary

- **R2R**: The retrieval engine at port 7272. Stores documents as vector embeddings, answers RAG queries, and holds conversation message history. Accessed via the R2R Python SDK; never exposed directly to the frontend.
- **R2R collection**: A named grouping in R2R that filters which documents are searched during retrieval. Each Notebook gets exactly one, passed as a `collection_ids` filter in `search_settings`.
- **R2R conversation ID (`r2r_conv_id`)**: An opaque identifier R2R assigns to a conversation thread. The `conversations` SQLite table stores this alongside local metadata; R2R remains the store for actual messages.
- **RAG (Retrieval-Augmented Generation)**: A pattern where retrieved document chunks are injected as context before an LLM generates an answer. This system uses R2R for retrieval and Gemini for generation.
- **DocQuery**: This codebase's project name. Also appears as `docquery_document_id` — the local document identifier assigned during ingestion, distinct from R2R's internal IDs.
- **Gemini Flash (`gemini-3-flash-preview`)**: The Google Gemini model used for all LLM calls — structure generation, page synthesis, R2R completions, and RAGAS evaluation.
- **FCIS (Functional Core / Imperative Shell)**: A project-wide architecture pattern. Every Python module is labeled FC (pure functions, no I/O, fully unit-testable) or IS (orchestrates I/O, tested with mocks). Required by CLAUDE.md; each new file carries a `# pattern:` comment.
- **BackgroundTask**: A FastAPI primitive that runs a coroutine after the HTTP response is sent. Used so wiki generation can return 202 immediately and continue generating in the background.
- **Lifespan hook**: A FastAPI application startup/shutdown hook. `notebook_store.migrate_default_notebook()` is added here to run once on API startup before any request is served.
- **Default Notebook**: A synthetic notebook created during the startup migration to own all documents that existed before notebooks were introduced. Ensures backward compatibility on upgrade.
- **nb-ui**: The prior design phase that established existing UI conventions (shadcn/ui, SSE streaming, App Router structure) that this design follows.
- **App Router**: Next.js 13+ routing model used in `apps/web/`. Routes live under `app/` using server/client component conventions.
- **`next/dynamic({ ssr: false })`**: A Next.js import wrapper that defers a component's loading to the browser only. Required for `MermaidDiagram` because Mermaid manipulates the DOM and cannot run server-side.
- **Mermaid**: A JavaScript diagramming library that renders flowcharts, sequence diagrams, and class diagrams from plain-text syntax embedded in Markdown code blocks.
- **deepwiki-open**: An open-source wiki-generation project whose XML structure schema and Mermaid diagram instruction prompts this design borrows.
- **LangGraph**: A graph-based orchestration framework for LLM workflows. Explicitly out of scope for this design — two-step async generation does not require it.
- **WikiStructure / WikiPageSpec**: Internal domain types. `WikiStructure` holds the parsed outline (sections → page slugs); `WikiPageSpec` is the per-page descriptor (slug, title, `source_doc_ids`, importance) used to drive parallel page generation.
- **`INSERT OR REPLACE` (UPSERT)**: A SQLite operation used when regenerating a wiki. Existing rows matching `UNIQUE(notebook_id, slug)` are atomically replaced without creating orphaned rows.
- **`asyncio.gather`**: Python standard library call for running multiple coroutines concurrently. Used in `wiki_generator.py` to issue N page-synthesis LLM calls in parallel.

## Architecture

Stack: FastAPI :8000 + Next.js :3000 App Router + R2R :7272 + SQLite + Gemini Flash. FCIS pattern on all Python modules.

Service topology is unchanged from nb-ui:

```
PDFs → ingestion pipeline → R2R :7272 → apps/api :8000 → apps/web :3000
```

### Data stores

Three new SQLite databases following the `data/*.db` convention:

**`data/notebooks.db`**
```sql
notebooks (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT,
  r2r_collection_id TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
)

notebook_documents (
  notebook_id TEXT NOT NULL REFERENCES notebooks(id),
  document_id TEXT NOT NULL,
  added_at TEXT NOT NULL,
  PRIMARY KEY (notebook_id, document_id)
)
```

**`data/wiki.db`**
```sql
wiki_jobs (
  job_id TEXT PRIMARY KEY,
  notebook_id TEXT NOT NULL,
  status TEXT NOT NULL,      -- queued | running | completed | failed
  pages_done INTEGER DEFAULT 0,
  pages_total INTEGER DEFAULT 0,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  error TEXT
)

wiki_structure (
  notebook_id TEXT PRIMARY KEY,
  title TEXT NOT NULL,
  description TEXT,
  sections_json TEXT NOT NULL,  -- JSON: [{id, title, page_slugs[]}]
  generated_at TEXT NOT NULL
)

wiki_pages (
  id TEXT PRIMARY KEY,
  notebook_id TEXT NOT NULL,
  slug TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT,
  content TEXT,               -- markdown, null while generating
  importance TEXT NOT NULL,   -- high | medium | low
  source_doc_ids_json TEXT,   -- JSON: [docquery_document_id, ...]
  related_slugs_json TEXT,    -- JSON: [slug, ...]
  generated_at TEXT,
  UNIQUE(notebook_id, slug)
)
```

**`data/conversations.db`**
```sql
conversations (
  id TEXT PRIMARY KEY,
  notebook_id TEXT NOT NULL,
  r2r_conv_id TEXT NOT NULL,
  title TEXT,                 -- auto-generated from first message, max 60 chars
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
)
```

### R2R collection isolation

Each Notebook maps 1:1 to an R2R collection:

- **Create notebook** → `client.collections.create(name=name)` → store `r2r_collection_id`
- **Ingest document** → `client.documents.create(chunks=..., collection_ids=[r2r_collection_id])`
- **RAG query** → `search_settings` gains `filters: {"collection_ids": {"$in": [r2r_collection_id]}}`
- **Delete notebook** → `client.collections.delete(collection_id)` — documents in R2R are NOT deleted

SQLite `notebook_documents` is the source of truth for membership. R2R collection is kept in sync as a mirror; if they diverge, SQLite wins.

### Wiki generation

Two-step pipeline, no LangGraph:

**Step 1 — Structure** (one LLM call): build a doc manifest (titles, page counts, doc IDs for each document in the notebook) → send to Gemini with the structure prompt → parse XML response → store `wiki_structure` + empty `wiki_pages` rows.

**Step 2 — Pages** (N parallel LLM calls): for each `wiki_page`, retrieve top-K chunks from R2R using the page's `source_doc_ids` as a collection filter → send to Gemini with the page prompt → store markdown content + update `wiki_jobs.pages_done`.

Both steps run inside a FastAPI `BackgroundTask`. Progress is tracked in `wiki_jobs` and polled by the frontend.

Wiki structure prompt borrows the XML schema from deepwiki-open. Page prompt borrows deepwiki-open's Mermaid diagram instructions (graph TD, sequenceDiagram, classDiagram with strict syntax rules).

### Conversation persistence

R2R holds message history (unchanged). The new `conversations` SQLite table stores only metadata: our UUID, the R2R conversation ID, the notebook it belongs to, and an auto-generated title (first message text, truncated to 60 chars — no LLM call needed).

### API surface


## Existing Patterns

This design follows all patterns established in nb-ui and the existing API layer:

- **FCIS:** All new Python modules carry `# pattern: Functional Core` or `# pattern: Imperative Shell`. Pure helpers in FC files; I/O orchestrators in IS files.
- **SQLite store pattern:** `threading.Lock` + `_ensure_table` + lazy `_connect()` that mkdirs the parent. Copied from `apps/api/services/ingest_job_store.py`.
- **Job status machine:** `queued → running → completed | failed`. Startup sweep flips stale `running` jobs to `failed`. Copied from `apps/api/services/ingest_jobs.py`.
- **Lifespan hook:** Idempotent setup call before `yield` in `apps/api/main.py`. Migration hook follows the same pattern as `ingest_job_store.mark_stale_running_jobs()`.
- **R2R client:** `get_client()` factory in `r2r_client.py`; `DEFAULT_SEARCH_SETTINGS` extended with per-call overrides; SDK never imported in routes.
- **Route structure:** FastAPI routers in `apps/api/routes/`; services in `apps/api/services/`.
- **Frontend:** shadcn/ui components, SSE streaming via native `EventSource`, App Router pages under `apps/web/app/(app)/`.

## Additional Considerations

**Mermaid rendering:** Must be client-only. `MermaidDiagram` uses `next/dynamic({ ssr: false })`. Initialize once via `mermaid.initialize({ startOnLoad: false })` and call `mermaid.run()` per render.

**Wiki XML parser robustness:** `parse_wiki_structure_xml()` must not raise on malformed LLM output. Return an empty `WikiStructure` with a single "Overview" page as fallback. Test with intentionally truncated and invalid XML.

**R2R document retention on notebook delete:** Deleting a notebook removes the R2R collection association and all local SQLite records but does NOT delete R2R documents. Documents remain in R2R storage and can be re-assigned if needed.

**Default Notebook migration idempotency:** `migrate_default_notebook()` checks for existing "Default Notebook" row before creating. Safe to call on every startup.

**Wiki page UPSERT:** `wiki_pages` has `UNIQUE(notebook_id, slug)`. On regeneration, use `INSERT OR REPLACE` so existing page content is atomically replaced without leaving orphaned rows.

**Implementation scoping:** 8 phases split across 2 implementation runs. Run 1 (phases 1–4) delivers the notebook model end-to-end. Run 2 (phases 5–8) delivers wiki and conversation history. The writing-plans skill limit of 8 phases per run is respected.
