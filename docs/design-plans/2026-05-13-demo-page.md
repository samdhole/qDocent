# Demo Page Design

## Summary

This document specifies a demo page (`/demo`) for the DocQuery RAG portfolio application that replaces the Evaluations nav entry with a showcase of the system's core capabilities against a real public 10-K filing. The page matters commercially because it lets reviewers experience the full ingestion-to-RAG pipeline — wiki generation, cited answers, and figure matching — without needing a running backend, which lowers the barrier for async portfolio evaluation. The key architectural choice is a clean Server Component / Client Component split following the FCIS pattern: three static showcase panels (`WikiPreviewPanel`, `QAShowcasePanel`, `FigurePanel`) are pure Server Components that import committed JSON snapshots at build time with zero runtime API calls, while `DemoAskBox` is an Imperative Shell client component that health-checks the API on mount and degrades gracefully to the same pre-baked snapshot when the backend is unavailable. All snapshot data is generated once via `make demo-setup` — a fully automated script that downloads a public SEC EDGAR filing, ingests it, generates a wiki, and captures representative output files — and then committed to the repository so the static panels work without any services running.

## Definition of Done

1. **Demo page replaces Evaluations in the nav** — new route `/demo`, sidebar shows "Demo" not "Evaluations". RAGAS table removed from UI (eval still runs via `make eval`, just not surfaced in the nav).

2. **Demo page has three static showcase panels** (always visible, no API required):
   - Wiki structure preview — rendered from `data/demo/wiki_structure.json` using existing wiki components
   - Example Q&A with citation — rendered from `data/demo/example_qa.json` using `AnswerCard`
   - Figure match panel — static `data/demo/example_figure.png` + caption

3. **Live "Try it" ask box** — queries the demo notebook when R2R is running; shows the pre-baked Q&A answer gracefully when API is down.

4. **Robinhood or Affirm 10-K ingested as the demo corpus** — downloaded from SEC EDGAR (public filing), ingested into a "Demo" notebook, wiki generated, snapshots captured and committed to repo. (Stripe S-1 is not publicly available — Stripe has not IPO'd.)

## Acceptance Criteria

### demo-page.AC1: Nav and routing

- **demo-page.AC1.1 Success:** "Demo" appears in the sidebar nav; "Evaluations" does not
- **demo-page.AC1.2 Success:** Navigating to `/demo` renders the demo page without errors
- **demo-page.AC1.3 Success:** `/evals` route is still accessible by direct URL and renders the RAGAS table
- **demo-page.AC1.4 Failure:** No 404 or runtime error on first load of `/demo` with stub data

### demo-page.AC2: Static showcase panels

- **demo-page.AC2.1 Success:** WikiPreviewPanel renders at least one section title from `wiki_structure.json` without making any API call
- **demo-page.AC2.2 Success:** QAShowcasePanel renders the answer text and at least one citation badge from `example_qa.json` without making any API call
- **demo-page.AC2.3 Success:** FigurePanel renders the example figure image and a non-empty caption
- **demo-page.AC2.4 Failure:** If `wiki_structure.json` or `example_qa.json` is malformed, the build fails (TypeScript cast catches shape mismatch at compile time, not runtime)

### demo-page.AC3: DemoAskBox degradation

- **demo-page.AC3.1 Success:** When `GET /health` returns 200, `ConversationView` is rendered with the demo notebook scoped
- **demo-page.AC3.2 Success:** When `GET /health` times out or errors, the pre-baked `AnswerCard` from `example_qa.json` is shown with the "showing cached example" badge
- **demo-page.AC3.3 Success:** `NEXT_PUBLIC_DEMO_NOTEBOOK_ID` unset → DemoAskBox renders cached answer (does not crash)
- **demo-page.AC3.4 Edge:** Health check uses a 2-second timeout; does not block page render (async after mount)

### demo-page.AC4: Demo corpus snapshots

- **demo-page.AC4.1 Success:** `make demo-setup` completes end-to-end and writes all three data files
- **demo-page.AC4.2 Success:** `make demo-setup` is idempotent — running it twice does not create duplicate notebooks or overwrite existing snapshot files
- **demo-page.AC4.3 Success:** Committed snapshot files contain real fintech 10-K content (not stub placeholders)

## Glossary

**DemoAskBox** — a `"use client"` React component that fires a 2-second health-check against the FastAPI `/health` endpoint on mount, then either renders a live `ConversationView` scoped to the demo notebook or falls back to a pre-baked `AnswerCard` with a "showing cached example" badge.

**WikiPreviewPanel** — a Next.js Server Component that imports `wiki_structure.json` at build time and renders the wiki section tree and first-page excerpt using the existing `WikiTreeNav` and `WikiPage` components without making any API calls.

**QAShowcasePanel** — a Next.js Server Component that imports `example_qa.json` at build time and renders an `AnswerCard` with citation badges in hover-only mode (no SourcePanel interaction) to illustrate a cited RAG answer.

**FigurePanel** — an inline Server Component section in `page.tsx` that renders a static `example_figure.png` image and a caption, demonstrating the figure-matching capability of the ingestion pipeline.

**AskResponse** — the TypeScript interface in `apps/web/lib/types.ts` that describes the shape of a RAG answer: `{ question, answer, citations, retrieved_contexts, figures, confidence_label, needs_human_review }`. `example_qa.json` must conform to this shape.

**WikiStructure** — the TypeScript interface consumed by `WikiTreeNav` that describes a generated wiki: `{ title, sections: Array<{ title, pages: Array<{ slug, title }> }> }`. `wiki_structure.json` must conform to this shape.

**Server Component** — a Next.js React component that renders exclusively on the server (or at build time); it can import JSON files and perform synchronous data access but cannot use browser APIs, hooks, or event handlers. Equivalent to the Functional Core in the FCIS pattern for this project.

**FCIS** — Functional Core / Imperative Shell, the architectural pattern mandated in this codebase: pure, side-effect-free modules (FC) are tested directly, while I/O-performing orchestrators (IS) are mocked at their boundary. Server Components map to FC; client components performing fetches map to IS.

**NEXT_PUBLIC_DEMO_NOTEBOOK_ID** — a Next.js build-time environment variable that holds the UUID of the pre-ingested "Demo" notebook in R2R; if unset, `DemoAskBox` skips the live path and renders the cached answer only.

**make demo-setup** — a Makefile target that runs `scripts/setup_demo_corpus.py`, which downloads a public SEC EDGAR 10-K PDF, ingests it into a "Demo" notebook, generates a wiki, and writes the three committed snapshot files (`wiki_structure.json`, `example_qa.json`, `example_figure.png`); the script is idempotent and prints the resulting notebook ID.

**RAGAS** — Retrieval-Augmented Generation Assessment, the evaluation framework used in `packages/evals/` to score RAG pipeline outputs on metrics such as faithfulness and answer relevancy; still runnable via `make eval` but removed from the sidebar nav by this design.

**R2R** — the open-source retrieval engine (SciPhi R2R) running on port 7272 that handles vector storage, semantic search, and RAG response generation; all DocQuery retrieval routes through the FastAPI wrapper at port 8000 rather than directly to R2R.

## Architecture

The `/evals` nav entry is replaced with `/demo`. The existing `/evals` route stays intact (accessible by direct URL; `make eval` and `GET /eval/results` are unaffected) but is removed from `AppSidebar.tsx`'s `NAV_ITEMS`.

`app/(app)/demo/page.tsx` is a **Server Component** that imports two JSON snapshot files at build time (no runtime API calls for static content). It renders four sections:

```
DemoPage (Server Component)
  ├── WikiPreviewPanel     — section tree + first page excerpt (server)
  ├── QAShowcasePanel      — AnswerCard with example citation (server)
  ├── FigurePanel          — static image + caption (server, inline JSX)
  └── DemoAskBox           — live ask scoped to demo notebook ("use client")
```

**Data files committed to repo:**

```
apps/web/app/(app)/demo/data/
  wiki_structure.json   ← WikiStructure shape
  example_qa.json       ← AskResponse shape
apps/web/public/demo/
  example_figure.png    ← served at /demo/example_figure.png by Next.js
```

**DemoAskBox degradation flow:**

```
mount → GET /health (2 s timeout)
  ├── 200 OK  → ConversationView with notebookId=NEXT_PUBLIC_DEMO_NOTEBOOK_ID
  └── error   → AnswerCard from example_qa.json
               + Badge: "Live demo unavailable — showing cached example"
```

`NEXT_PUBLIC_DEMO_NOTEBOOK_ID` is added to `.env.example` and read as a build-time env var. If unset, `DemoAskBox` renders the cached answer only.

**Snapshot contracts:**

`wiki_structure.json` must satisfy `WikiStructure` from `apps/web/components/WikiTreeNav.tsx`:
```ts
{ title: string; sections: Array<{ title: string; pages: Array<{ slug: string; title: string }> }> }
```

`example_qa.json` must satisfy `AskResponse` from `apps/web/lib/types.ts`:
```ts
{ question: string; answer: string; citations: Citation[]; retrieved_contexts: RetrievedContext[];
  figures: FigureAsset[]; confidence_label: string; needs_human_review: boolean }
```

## Existing Patterns

**AppSidebar NAV_ITEMS** (`apps/web/components/AppSidebar.tsx`, line 13–20): array of `{ href, label, icon }` objects. Active route detected via `pathname.startsWith(href)`. New `/demo` entry follows identical shape using `Sparkles` from `lucide-react`.

**WikiPage / WikiTreeNav** (`apps/web/components/WikiPage.tsx`, `WikiTreeNav.tsx`): `WikiPreviewPanel` reuses both components without modification. `WikiPage` receives `{ title, content, sourceDocIds }` from the first page in the snapshot. `WikiTreeNav` receives `{ structure: WikiStructure, notebookId: string }` — for the static panel `notebookId` is the demo notebook ID (links in the tree will be live when API is up).

**AnswerCard** (`apps/web/components/AnswerCard.tsx`): `QAShowcasePanel` passes the snapshot directly. `onSelectCitation` is omitted — citation badges render in hover-only mode (no SourcePanel opens), which is intentional for a showcase panel.

**ConversationView** (`apps/web/components/ConversationView.tsx`): `DemoAskBox` wraps this component with `notebookId` prop to scope retrieval to the demo notebook's R2R collection. This is an existing prop, no modification required.

**FCIS pattern**: Server Components (`WikiPreviewPanel`, `QAShowcasePanel`, `FigurePanel`) are Functional Core — pure render from imported data. `DemoAskBox` is Imperative Shell — performs I/O (health check fetch, conversation stream). New Python script follows `# pattern: Imperative Shell`.

**Static file serving**: `apps/web/public/` is the established Next.js static directory. `example_figure.png` is placed there and served at `/demo/example_figure.png` — same pattern used by the existing scaffold SVGs.

### Phase 4: Demo Corpus Setup Script

**Goal:** Automate one-time ingestion of a public fintech 10-K and snapshot capture.

**Components:**
- `scripts/setup_demo_corpus.py` — `# pattern: Imperative Shell`; steps: (1) download Robinhood or Affirm 10-K PDF from SEC EDGAR public URL, (2) `POST /notebooks` to create "Demo" notebook, (3) `POST /notebooks/{id}/documents` to upload PDF, poll until complete, (4) `POST /notebooks/{id}/wiki/generate`, poll until `completed`, (5) `GET /notebooks/{id}/wiki` → write `apps/web/app/(app)/demo/data/wiki_structure.json`, (6) `POST /ask` with scoped demo question → write `apps/web/app/(app)/demo/data/example_qa.json`, (7) copy first extracted figure to `apps/web/public/demo/example_figure.png`; idempotent (skip steps if output files already exist); prints `DEMO_NOTEBOOK_ID=<uuid>` at end

**Dependencies:** Phase 3 (env var placeholder exists); running API + R2R

**Done when:** `make demo-setup` completes end-to-end against a running stack; three data files are written with real content; notebook ID is printed

## Additional Considerations

**RAGAS eval preservation:** The `/evals` route (`app/(app)/evals/page.tsx`, `components/EvalTable.tsx`) is untouched. `GET /eval/results` and `POST /eval/run` backend routes are untouched. `make eval` continues to work. The only change is removal from `NAV_ITEMS`.

**WikiTreeNav links in static panel:** `WikiTreeNav` renders links to `/notebooks/{notebookId}/wiki/{slug}`. In the static `WikiPreviewPanel`, these links will be live URLs — they work when R2R is up and the demo notebook exists, and return 404 when not. This is acceptable for a demo page (not broken, just not navigable in offline mode).

**JSON import type safety:** TypeScript `resolveJsonModule` must be enabled (default in Next.js). The imports `import wikiSnapshot from './data/wiki_structure.json'` and `import qaSnapshot from './data/example_qa.json'` will be typed as `any` unless `as WikiStructure` / `as AskResponse` casts are added in the consuming component — cast at the import site.
