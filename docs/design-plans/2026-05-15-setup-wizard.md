# Setup Wizard Design

## Summary

This design introduces a first-run setup wizard and a daily launcher for the DocQuery Docker stack, targeting non-technical clients on Windows and Linux. The problem it solves is the manual bootstrap gap: today a client who receives the repo must hand-edit two sensitive config files (`.env` and `r2r_gemini.toml`), run a WSL ownership fix, and figure out the right `docker compose` incantation on their own. The wizard collapses that into a single script that asks two questions — a Google API key and an R2R admin password — and handles the rest automatically.

The approach is intentionally minimal: four plain shell scripts (`setup.ps1`, `setup.sh`, `start.ps1`, `start.sh`) in a new `install/` directory, with no installer framework, no Python, and no Node required before Docker is available. Config files are generated by substituting sentinel values directly in the provided example files using `sed` (bash) and PowerShell's `-replace` operator. The start scripts serve as a daily driver — idempotent enough to open the browser immediately if the stack is already up, or bring it up and wait for a health check before launching if not. Implementation is staged: start scripts first (simpler, prove the health-wait pattern), setup scripts second, documentation updates last.

## Definition of Done

- `install/setup.ps1` (Windows) and `install/setup.sh` (Linux) prompt for `GOOGLE_API_KEY` and R2R admin password, write `.env` and `r2r_gemini.toml`, run the WSL chown step, build the Docker images, start the stack, and open `http://localhost:3000` in the browser.
- `install/start.ps1` (Windows) and `install/start.sh` (Linux) run `docker compose up -d`, wait for healthy status, then open the browser — if already running, just open the browser.
- A non-technical client on Windows can get from zero to working app by running one script and answering two prompts.

## Acceptance Criteria

### setup-wizard.AC1: setup script configures the environment
- **setup-wizard.AC1.1 Success:** Running `setup.ps1` / `setup.sh` on a fresh clone with valid inputs produces a correct `.env` (`GOOGLE_API_KEY` set) and `r2r_gemini.toml` (email + password substituted, no `CHANGE_ME` sentinel remaining)
- **setup-wizard.AC1.2 Success:** Password confirmation loop re-prompts on mismatch without writing any files
- **setup-wizard.AC1.3 Failure:** Passwords shorter than 8 characters are rejected with a clear message
- **setup-wizard.AC1.4 Failure:** If Docker is not running, script exits with a non-zero code and prints an install/start hint
- **setup-wizard.AC1.5 Edge:** If `.env` or `r2r_gemini.toml` already exist, script warns and prompts before overwriting; default answer is No

### setup-wizard.AC2: setup script starts the stack and opens the browser
- **setup-wizard.AC2.1 Success:** After config writes, `docker compose build` and `up -d` run and the browser opens to `http://localhost:3000`
- **setup-wizard.AC2.2 Failure:** If health check does not pass within 60 s, script prints last 20 lines of `docker compose logs api` and exits 1
- **setup-wizard.AC2.3 Success (Windows):** WSL chown step runs before `docker compose build`

### setup-wizard.AC3: start script launches or joins a running stack
- **setup-wizard.AC3.1 Success:** If stack is not running, `start.ps1` / `start.sh` runs `docker compose up -d`, waits for healthy, and opens `http://localhost:3000`
- **setup-wizard.AC3.2 Success:** If stack is already running, script skips `docker compose up -d` and opens the browser directly
- **setup-wizard.AC3.3 Failure:** If `.env` is missing, script exits 1 with "run setup first" message
- **setup-wizard.AC3.4 Failure:** If Docker is not running, script exits 1 with a clear message
- **setup-wizard.AC3.5 Failure:** If health check times out, script prints `docker compose logs api` tail and exits 1

### setup-wizard.AC4: scripts are structurally sound
- **setup-wizard.AC4.1:** `install/setup.sh` and `install/start.sh` pass `bash -n` syntax check
- **setup-wizard.AC4.2:** `install/setup.ps1` and `install/start.ps1` pass `pwsh` syntax check
- **setup-wizard.AC4.3:** `.sh` scripts have executable bit set and LF line endings
- **setup-wizard.AC4.4:** `install/README.md` covers both Windows and Linux quickstart paths

## Glossary

- **R2R**: The open-source retrieval and RAG server (by SciPhi-AI) that this application wraps. Handles vector storage, document ingest, and LLM-backed question answering. Configured via `r2r_gemini.toml`.
- **`r2r_gemini.toml`**: The project-local R2R configuration file that wires all LLM slots to Google Gemini via litellm. Copied from `r2r_gemini.toml.example`; never committed because it contains the admin password.
- **`.env` / `.env.example.client`**: Environment variable files consumed by Docker Compose. `.env.example.client` is the trimmed handoff template; setup scripts copy it and substitute the API key.
- **sentinel value**: A placeholder string in a template file (e.g. `CHANGE_ME`, a blank `GOOGLE_API_KEY=`) that scripts replace with real values using `sed` or PowerShell `-replace`. No template language needed.
- **WSL (Windows Subsystem for Linux)**: Microsoft's Linux compatibility layer, required by Docker Desktop on Windows. The setup script calls `wsl sudo chown` to fix data-directory ownership so Linux containers can write to the host-mounted `data/` folder.
- **chown step**: `chown -R 1001:1001 data` — sets the `data/` directory owner to UID/GID 1001, matching the non-root user inside the API container. Required on both Windows (via WSL) and Linux.
- **health wait loop**: A polling loop that hits `http://localhost:8000/health` every 3 seconds for up to 60 seconds, confirming the API container is accepting requests before the browser is opened.
- **`docker compose ps`**: Docker Compose subcommand used by the start scripts to check whether containers are already running, enabling idempotent behaviour.
- **`xdg-open`**: Standard Linux utility for opening a URL in the default browser. Bash scripts use it with a fallback to `open` (macOS).
- **`Start-Process`**: PowerShell cmdlet used to open a URL in the default browser on Windows.
- **`Read-Host -AsSecureString`**: PowerShell's masked input prompt — shows asterisks instead of characters, used for API key and password prompts.
- **`read -s`**: Bash flag for silent/masked terminal input, same purpose as above in `.sh` scripts.
- **`bash -n`**: Bash's syntax-check-only mode; test suite runs scripts through this to catch parse errors without executing them.
- **`pwsh`**: PowerShell Core (version 6+), used in CI syntax checks alongside the Windows PowerShell 5.1 target runtime.
- **LF line endings**: Unix-style line endings (`\n` only). Required for `.sh` scripts to be correctly parsed in Linux containers and WSL, even when checked out on Windows.
- **overwrite guard**: The prompt shown when `.env` or `r2r_gemini.toml` already exist, defaulting to "No" to prevent accidental data loss on a second run of setup.
- **Docker Compose healthcheck**: Docker-native mechanism that marks a container `healthy` only after a specified probe succeeds; scripts depend on this status for the API container before opening the browser.

## Architecture

Four plain shell scripts in `install/` — no installer framework, no Python, no Node required before Docker is installed.

```
install/
  setup.ps1      # Windows one-time setup wizard
  setup.sh       # Linux one-time setup wizard
  start.ps1      # Windows daily launcher
  start.sh       # Linux daily launcher
  README.md      # One-paragraph quickstart
```

**setup scripts** are the first thing a client runs on a fresh clone. They are interactive wizards: prompt → write config files → build images → start stack → open browser. After setup completes, the client never needs to run setup again.

**start scripts** are the daily driver. The client double-clicks (or runs) `start.ps1` / `start.sh` each time they want to use the app. If the stack is already running, they just open the browser. If not, they bring it up and wait for health.

Scripts have no runtime dependencies beyond Docker Desktop (Windows) or Docker Engine (Linux) and a POSIX shell / PowerShell 5.1+.

---

**setup flow (both platforms):**

1. Print welcome banner
2. Check Docker is installed and running — exit 1 with install link if not
3. Prompt: `GOOGLE_API_KEY` (masked input)
4. Prompt: R2R admin email (default `admin@example.com`, Enter to accept)
5. Prompt: R2R admin password (masked, confirm twice, min 8 chars)
6. If `.env` or `r2r_gemini.toml` already exist → warn and prompt before overwriting
7. Write `.env` — copy `.env.example.client`, substitute `GOOGLE_API_KEY`
8. Write `r2r_gemini.toml` — copy `r2r_gemini.toml.example`, substitute email + password
9. Platform-specific chown:
   - Windows: `wsl mkdir -p data && wsl sudo chown -R 1001:1001 data`
   - Linux: `sudo mkdir -p data && sudo chown -R 1001:1001 data`
10. `docker compose build` (output visible)
11. `docker compose up -d`
12. Health wait loop: poll `http://localhost:8000/health` every 3 s, up to 60 s
13. Open browser to `http://localhost:3000`
14. Print completion message with start script reminder

---

**start flow (both platforms):**

1. Check Docker is running — exit 1 if not
2. Check `.env` exists — exit 1 with "run setup first" if missing
3. Check if api container is already running (`docker compose ps`)
4. If running → skip to step 6
5. `docker compose up -d` → health wait loop (3 s poll, 60 s timeout)
6. Open `http://localhost:3000` in default browser
7. On health timeout → print last 20 lines of `docker compose logs api`, exit 1

---

**Config substitution approach:** Scripts use `sed` (bash) and `-replace` (PowerShell) to substitute sentinel values in the example files. The sentinel values are `GOOGLE_API_KEY=` (blank, in `.env.example.client`) and `CHANGE_ME` (in `r2r_gemini.toml.example`). No template language needed.

**Browser open:** PowerShell uses `Start-Process`; bash uses `xdg-open` (Linux) with fallback to `open` (macOS, for future use).

## Existing Patterns

Investigation found no existing shell installer scripts in the repo. The `scripts/` directory contains only Python dev tools (`smoke_r2r.py`, `ingest_sample_docs.py`, etc.) which require the Python venv already set up — they are not suitable as client-facing entry points.

This design introduces new patterns to the repo:
- Interactive shell scripts with user prompts (`install/`)
- Config file generation via sentinel substitution

These scripts deliberately avoid reusing `scripts/` to keep Python dev tooling separate from Docker client tooling.

## Additional Considerations

**Password minimum length:** The setup scripts enforce minimum 8 characters on the R2R admin password (confirm loop re-prompts on mismatch or too-short input). This matches R2R's own validation — a password rejected by R2R after a 5-minute build would be a bad client experience.

**Idempotent overwrite guard:** If `.env` or `r2r_gemini.toml` already exist, the setup script warns and asks `Overwrite? [y/N]`. Default is No. This prevents accidental data loss on a second run.

**WSL availability on Windows:** The chown step calls `wsl`. If WSL is not installed (rare but possible on older Windows 10 machines without WSL2), the script prints a manual fallback instruction rather than failing silently. Docker Desktop's WSL2 requirement means WSL must be present for Docker to work at all — so this is an edge case, not the normal path.
